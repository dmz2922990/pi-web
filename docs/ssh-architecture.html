Layer Overview

Layer 1 pi-agent SDK — Interfaces & Tool Factory

Defines 7 pluggable operation interfaces (Strategy Pattern). Each built-in tool (bash, read, write, edit, grep, find, ls) delegates all I/O to an injected operations object.

  • BashOperations — exec(command, cwd, options)
  • ReadOperations — readFile(), access(), detectImageMimeType()
  • WriteOperations — writeFile(), mkdir()
  • EditOperations — readFile(), writeFile(), access()
  • GrepOperations — isDirectory(), readFile()
  • FindOperations — exists(), glob()
  • LsOperations — exists(), stat(), readdir()

Factory: createCodingTools(cwd, ToolsOptions) → returns tool instance array. When ToolsOptions.bash.operations = myOps, the bash tool calls myOps.exec() instead of local child_process.

Layer 2 pi-web — SSH Operations Implementation

ssh-operations.ts implements all 7 interfaces using SSH commands:

Interface MethodSSH Implementation
bash.exec()ssh2.Client.exec() shell command
read.readFile()ssh2 SFTP createReadStream
write.writeFile()echo BASE64 | base64 -d > file via shell (SFTP write hangs on some servers)
write.mkdir()mkdir -p via shell
find.glob()find <cwd> -name <pat> -type f | head
ls.*SFTP stat/readdir
*.exists()test -e via shell

ssh-tool-factory.ts bridges Layer 1 & 2: createSshTools(cwd, sshConn) → builds ToolsOptions with all SSH ops → calls createCodingTools().

Layer 3 pi-web — Orchestration (BubbleManager)

BubbleManager decides local vs remote and wires everything:

  • Reads hostSelections from user input → resolves HostConfig → converts to SshConfig
  • If executionMode === "ssh": creates SshConnection → createSshTools() → createAgentSession({ noTools: "builtin", customTools: sshTools })
  • If local: standard createAgentSession() with built-in tools
  • Manages SSH lifecycle (connect, reconnect, disconnect on destroy)
  • Persists hostId in bubble data for restart recovery
Layer 4 ssh2 — Transport Layer

ssh2 npm package provides the SSH protocol implementation (TCP → encrypted channel → shell/SFTP). SshConnection wraps it with:

  • Auto-reconnect on disconnect
  • Keepalive (15s interval)
  • Password and privateKey auth
  • 60s timeout on exec/read/write
  • Shell escape for paths with special chars

Responsibility Matrix

pi-agent SDK provides

  • Operations interfaces — 7 abstract contracts for I/O
  • ToolsOptions injection point — { bash: { operations: ... } }
  • createCodingTools() — tool factory accepting injected ops
  • createAgentSession() — session factory accepting custom tools
  • Agent runtime — LLM loop, tool calling, streaming
  • Tool logic — edit computes diff, grep does pattern matching, etc.

pi-web provides

  • SSH transport — SshConnection (connect, exec, SFTP, reconnect)
  • Operations impl — all 7 interfaces backed by SSH
  • Orchestration — BubbleManager decides local/remote per worker
  • Host management — CRUD for SSH host configs in UI
  • Session persistence — saves hostId, restores on restart
  • Bridge factory — createSshTools() wires ops → SDK

Key Design Insight

The agent is completely unaware it's running remotely.

pi-agent's tools never call fs or child_process directly — they call operations.readFile(), operations.exec(), etc. pi-web injects SSH-backed implementations. The LLM generates the same tool calls regardless of where execution happens.

This means: adding a new transport (Docker, Kubernetes, WebSocket) only requires implementing the 7 Operations interfaces — no changes to pi-agent or the agent's behavior.
